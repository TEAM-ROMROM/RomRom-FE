📝 현재 문제점
---

- `MainScreen`이 `IndexedStack`이라 5개 탭이 항상 메모리에 생존하며, 탭 전환 시 `initState`가 재실행되지 않는다. 그런데 공유 도메인 데이터를 각 화면이 로컬 `setState`로 들고 `initState`에서 1회만 로드해, 한 화면의 변경이 다른 화면에 반영되지 않는다.
- **대표 증상**: 내 물건을 등록/삭제/교환완료 처리한 뒤 홈 탭의 블러·카드 덱이 갱신되지 않는다(앱을 재시작해야 정상화). 신규 회원이 물건을 등록해도 홈에 "물건을 등록해 주세요" 안내가 계속 노출된다.
- 공유 도메인 상태가 13개+ 화면에 흩어져 있다.
  - 내 물건 목록(`getMyItems`): 홈 카드 덱·등록 탭·내 물건 탭·요청관리·프로필 교환 섹션 등 6개 화면이 각자 보관.
  - 회원 프로필(`getMemberInfo`): 8개 화면이 각자 독립 호출. 공유 소스 없음.
  - 채팅방 목록·알림 목록/미확인 수·요청 목록도 각 화면 로컬 보관.
- 갱신 전파 수단이 일관되지 않다.
  - 이벤트 버스(`AppEventBus` / `TradeCompletedEvent`)는 거래완료만 전파하고, 등록·삭제·상태변경은 전파 수단이 없어 타 화면이 stale된다.
  - 등록 탭은 같은 데이터를 들고도 이벤트를 구독하지 않아 거래완료 후에도 stale된다.
- 프로젝트 규칙(CLAUDE.md) 위반이 누적되어 있다.
  - `GlobalKey`로 다른 화면 메서드 직접 호출(등록 후 홈 갱신·코치마크 표시).
  - 목록에서 클라이언트가 수동으로 항목 제거(`removeWhere`) — "재조회만" 규칙 위반.
  - 알림 mute 설정이 두 화면에서 별도 store로 분열되어 서로 불일치.
- Riverpod는 좋아요·차단·알림설정 3개 기능에만 적용되어 있고, 목록·프로필 등 핵심 공유 상태는 여전히 raw `setState`다. 상태관리 표준 규칙이 없어 동일한 정합성 문제가 계속 재발한다.

🛠️ 해결 방안 / 제안 기능
---

- 모든 공유 도메인 상태를 Riverpod provider로 중앙화하여 단일 소유자를 두고, 화면은 구독만 한다.
- 이벤트 버스(`AppEventBus` / `TradeCompletedEvent`)와 `GlobalKey` 교차호출, 중복 store, 수동 목록 제거를 전부 제거한다.
- 4-레이어 표준 구조(repository → state → provider → Consumer 화면)를 정립한다. 비동기 목록은 `AsyncNotifier`, optimistic 토글은 동기 `Notifier`로 종류를 구분한다.
- mutation(등록·삭제·상태변경 등)은 화면이 아니라 provider의 메서드로만 수행하고, 내부에서 서버 재조회로 전체 화면을 자동 갱신한다.
- 정립한 상태관리 규칙을 CLAUDE.md와 `.claude/instructions/state-management.md`에 박제하여 이후 동일 문제 재발을 막는다.
- 설계·계획 문서
  - spec: `docs/superpowers/specs/2026-05-28-event-bus-to-riverpod-migration-design.md`
  - plan: `docs/superpowers/plans/2026-05-28-state-management-riverpod-centralization.md`
