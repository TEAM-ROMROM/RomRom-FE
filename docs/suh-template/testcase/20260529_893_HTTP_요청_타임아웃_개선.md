## 🧪 테스트 케이스

### TC-01: 일반 GET 요청 — 기본 타임아웃 15초 적용 확인
**목적**: 기본 타임아웃이 15초로 동작하는지 검증
**절차**:
1. 정상 네트워크에서 홈탭 진입
2. 카드 목록 API 응답 정상 수신 확인
3. 디버그 로그에서 응답 Duration이 정상 범위인지 확인

**기대 결과**: 정상 응답, timeout 미발생, 로그에 `Duration: NNNms` 표시

---

### TC-02: 일반 POST/PUT 요청 — 기본 타임아웃 15초 적용 확인
**목적**: JSON 본문 요청에서도 timeout 정상 동작
**절차**:
1. 회원 정보 수정 (PUT) 또는 좋아요 토글 (POST) 수행
2. 응답 정상 수신 확인

**기대 결과**: 정상 응답, switch문 제거 후에도 모든 HTTP 메소드 정상 동작

---

### TC-03: 이미지 업로드 — 60초 타임아웃 적용 확인
**목적**: 업로드 전용 60초 타임아웃 동작 검증
**절차**:
1. 마이페이지 → 프로필 이미지 변경 또는 물품 등록 화면 진입
2. 1MB 이상 이미지 5장 이상 선택
3. 업로드 진행 → 완료까지 대기

**기대 결과**:
- 정상 네트워크: 60초 내 업로드 완료
- 10초 초과 응답에도 timeout 발생하지 않음 (기존 10초 하드코딩이었으면 실패할 케이스)

---

### TC-04: 물품 등록 — 60초 타임아웃 적용 확인
**목적**: 물품 등록 API의 60초 타임아웃 동작
**절차**:
1. 물품 등록 화면 진입
2. 이미지 5장 이상 + 모든 필드 입력
3. 등록 버튼 탭

**기대 결과**: 정상 등록 완료. 등록 중 10초 초과해도 timeout 발생하지 않음

---

### TC-05: 약한 네트워크 — 헤더 수신 단계 타임아웃
**목적**: 서버 응답 헤더 자체가 늦어지는 경우 timeout
**절차**:
1. Charles Proxy 또는 dev tools로 네트워크 throttle (3G slow)
2. 일반 GET 요청 (홈 카드 로딩 등) 수행
3. 15초 초과 시 동작 확인

**기대 결과**:
- 디버그 로그에 `[Timeout] ... 15s 초과` 표시
- `TimeoutException` 발생, 사용자에게 적절한 에러 처리

---

### TC-06: 약한 네트워크 — body read 구간 타임아웃 (핵심 회귀 케이스)
**목적**: 헤더는 빨리 받았으나 body 수신이 느린 경우에도 timeout 동작 확인 (수정 핵심)
**절차**:
1. Charles Proxy로 body 응답만 지연 (throttle response)
2. 일반 요청 수행
3. 15초(또는 지정 timeout) 초과 시 동작 확인

**기대 결과**:
- 디버그 로그에 `[Timeout] ... Ns 초과 — body read` 표시
- 무한 hang 없이 정상적으로 TimeoutException 발생
- **수정 전엔 body read에서 hang됐을 케이스**

---

### TC-07: 이미지 업로드 약한 네트워크 — body read 60초 deadline
**목적**: 업로드 60초 timeout이 send+body read 전 구간에 적용되는지 확인
**절차**:
1. Charles로 throttle (slow 3G 등)
2. 5장 이미지 업로드 시도
3. 60초 초과 시 동작 확인

**기대 결과**: 60초 초과 시 `[Timeout] ... 60s 초과` 로그 출력 + TimeoutException

---

### TC-08: 토큰 만료 재시도 — timeout 파라미터 승계
**목적**: 401 후 토큰 갱신 재시도에서도 timeout 파라미터 유지 확인
**절차**:
1. 액세스 토큰 만료된 상태에서 이미지 업로드 시도
2. 토큰 자동 갱신 후 재요청 발생
3. 재요청에도 60초 timeout 적용되는지 로그 확인

**기대 결과**: 재요청에도 60초 timeout 유지 (15초 기본값으로 fallback 되지 않음)

---

### TC-09: 홈탭 카드 핸드 갱신 — didUpdateWidget cards 변경 반영 (관련 버그)
**목적**: 카드 목록 갱신 시 핸드 내부 상태 재생성 확인
**절차**:
1. 홈탭에서 카드 핸드 표시 상태
2. 새로고침 / 다른 탭 갔다 복귀 등으로 cards 리스트 변경 트리거
3. 카드 핸드가 새 cards 기준으로 다시 그려지는지 확인

**기대 결과**: 변경된 cards 기준으로 핸드 정상 갱신 (수정 전엔 stale 상태 유지됐을 케이스)

---

### TC-10: 제재 회원/UGC 위반 응답 — timeout 적용 후에도 정상 동작
**목적**: timeout 파라미터 추가가 기존 인증/제재 처리 로직에 영향 없는지
**절차**:
1. 제재된 계정 또는 UGC 위반 응답 트리거되는 요청 수행
2. 403 SUSPENDED_MEMBER / UGC violation 응답 처리 확인

**기대 결과**: 기존 제재/UGC 처리 흐름 정상 작동
