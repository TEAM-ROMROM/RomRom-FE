# 🚀[기능개선][API] HTTP 요청 타임아웃 개선 — 기본값 상향 및 업로드 전용 timeout 파라미터 도입

## 개요

`LoggingHttpClient`의 전역 타임아웃이 10초로 하드코딩되어 이미지 업로드·물품 등록처럼 처리 시간이 긴 요청에서 타임아웃 오류가 발생할 수 있었다. 기본값을 15초로 상향하고, 호출부에서 요청별 타임아웃을 직접 지정할 수 있는 `timeout` 파라미터를 공개 API 전 계층에 도입했다. 이미지 업로드(`/api/image/upload`)와 물품 등록/수정(`/api/item/post`, `/api/item/edit`)에는 60초를 명시했다.

## 변경 사항

### LoggingHttpClient — 타임아웃 추상화

- `lib/utils/log_http_client_interceptor.dart`: `defaultTimeout` 10초 → 15초 상향. 핵심 로직을 `sendWithTimeout(request, timeout)` 메서드로 분리하고, `send()`는 `sendWithTimeout(request, defaultTimeout)`으로 위임. `sendWithTimeout`은 타임아웃 초과 시 로그에 실제 설정값을 출력한다.

### ApiClient — timeout 파라미터 전파

- `lib/services/api_client.dart`: 공개 메서드 `sendMultipartRequest` / `sendHttpRequest`와 내부 메서드 `_executeMultipartRequest` / `_executeHttpRequest` 모두에 `Duration timeout = const Duration(seconds: 15)` 파라미터 추가. 토큰 갱신 후 재시도 시에도 동일한 timeout이 전달된다. `_executeHttpRequest`는 기존 단축 메서드(`get/post/...`) 대신 `http.Request`를 직접 빌드해 `sendWithTimeout`을 호출하도록 리팩토링했다.

### 업로드/등록 API — 60초 명시

- `lib/services/apis/image_api.dart`: `uploadImages` → `timeout: const Duration(seconds: 60)` 적용
- `lib/services/apis/item_api.dart`: `postItem`, `updateItem` → `timeout: const Duration(seconds: 60)` 적용

## 주요 구현 내용

기존 `send()` 메서드 내부에 타임아웃 값이 상수로 박혀 있어 호출부에서 조정이 불가능했다. 이를 해결하기 위해 두 계층으로 분리했다.

1. **LoggingHttpClient 수준**: `sendWithTimeout(request, timeout)`이 로깅·타임아웃·응답 재포장 전체를 담당. `send()`는 기본값(15초)을 넘겨 위임하므로 기존 동작은 그대로 유지된다.

2. **ApiClient 수준**: `sendMultipartRequest`→`_executeMultipartRequest`→`sendWithTimeout`, `sendHttpRequest`→`_executeHttpRequest`→`sendWithTimeout` 순으로 timeout이 파라미터로 흘러 내려간다. 외부에서 `timeout` 인자를 생략하면 기본 15초, 명시하면 그 값이 실제 소켓 타임아웃으로 적용된다.

## 주의사항

- 기존 호출부는 `timeout` 파라미터를 생략하면 되므로 **하위 호환성이 유지**된다.
- AI 가격 예측(`/api/item/price/predict`)은 ML 추론 시간이 불확실하므로 필요 시 호출부에서 별도 timeout 지정을 권장한다.
- `_refreshToken()`(토큰 재발급)은 기본 15초를 유지한다. 재발급 요청 자체가 오래 걸린다면 별도 조정이 필요하다.
