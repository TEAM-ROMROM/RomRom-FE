## 🧠 문제 정의

### 기존 동작
- `MainScreen`이 `IndexedStack` 구조라 5개 탭이 항상 메모리 상주 → `HomeTabScreen.initState`는 cold start 1회만 실행되고, **앱 재시작 전까지 피드가 절대 갱신되지 않음**.
- `_loadMoreItems`는 페이지 끝에 도달하면 `_currentPage = 0`으로 되감기되어 **완전히 동일한 아이템이 동일한 순서로 무한 반복** 노출.
- 이슈 #848("당겨서 새로고침")은 `PageView(vertical)`에 `RefreshIndicator` 직접 적용이 어렵고, 사용자가 능동으로 트리거해야 해서 발견율·사용성 모두 낮음.

### 핵심 의사결정
| 항목 | 결정 | 근거 |
|------|------|------|
| 트리거 | (a) 탭 재진입 + (b) 포그라운드 복귀 (홈일 때만) | "언제 최신화되는가?"를 사용자가 의식하지 않게 |
| Throttle | 직전 새로고침 후 **30초 이내** 스킵 | 탭 빠르게 왔다갔다할 때 API 폭격 방지 |
| 스크롤 위치 | 무조건 **page 0 jump** | 데이팅 앱 컨셉. 새 카드가 위에서부터 보여야 자연스러움 |
| 로딩 표시 | 상단 얇은 progress 바 | 자동 트리거라 풀스크린 스켈레톤은 거슬림. cold start만 풀스크린 |
| 중복 방지 | 본 itemId LRU 100개 → 응답 **끝으로 재정렬** (필터링 X) | seen 비율 높을 때 화면 비는 위험 차단 |
| 풀 소진 | 빈 페이지 응답 시 seen-set 자동 클리어 | 자연스럽게 다시 노출 |
| 서버 셔플 seed | 본 PR 범위 밖 (별도 BE 이슈로 분리) | FE 단독 출시 가능하게 격리 |

## ⚠️ 알려진 제약

- **세션 단위 seen-set**: 앱 재시작 시 seen-set은 비워짐. 영속 저장은 본 PR 범위 밖. 사용자가 같은 세션에서 본 것만 다양화 대상.
- **서버 셔플 미적용**: 클라 단독으로는 결국 같은 풀에서 노는 한계가 있어, 서버에 seed/cursor가 추가되어야 진짜 데이팅 앱식 다양화 완성. **별도 BE 이슈로 분리**됨.
- **자동 새로고침 silent fail**: 네트워크 에러 등 자동 새로고침 실패는 SnackBar로 surface하지 않음 — 사용자가 트리거한 게 아니라 의도. 디버그 로그만 남김.
- **iPad 영향 없음**: PageView · 광고 슬롯 · 카드 덱 모두 기존 레이아웃 그대로. iPad/대형기기 추가 검증 불필요.
